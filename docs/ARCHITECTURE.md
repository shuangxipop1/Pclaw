# Pclaw 架构设计文档

## 一、愿景与定位

### 终极定位

> **Pclaw = 人类与 Agent 协作的"责任桥梁"**
> - 核心理念：Agent 做事，人类确认，系统存证
> - 设计原则：最小核心 + 可组合 + 进化适应 + 开放生态

### 第一性原理

```
工程项目管理的本质 = 资源调配 + 任务协调 +责任归属

AI Agent 的本质 = 经验(训练) + 算力(运行) + 工具(扩展)

Pclaw 本质 = Agent协作平台 + 人类审核系统 + 责任存证系统
           = 让Agent替人类干活，人类为结果负责

核心问题：项目管理管什么？
答案：任务 + 人 + 责任
```

## 二、元能力 vs 分任务

### 元能力（核心三层 - 永不变）

```
┌─────────────────────────────────────────────────────┐
│                    元能力                             │
├─────────────────────────────────────────────────────┤
│  1️⃣ 任务系统                                        │
│     - 任务创建/分解                                   │
│     - 状态流转（待处理→进行中→待确认→完成）         │
│                                                      │
│  2️⃣ Agent调度                                       │
│     - 任务分配给Agent                                │
│     - 执行进度追踪                                   │
│     - 结果产出                                       │
│                                                      │
│  3️⃣ 确认与存证                                      │
│     - 人类审核（签字/邮件/系统）                    │
│     - 责任归属记录                                   │
│     - 审计日志                                       │
└─────────────────────────────────────────────────────┘
```

### 分任务（嵌套在元能力中 - 可选）

| 功能 | 嵌套位置 | 说明 |
|------|----------|------|
| 排期/依赖 | 任务属性 | 任务间有先后关系 |
| 多级审批 | 确认流 | 不同层级不同审批人 |
| 文件管理 | 任务产出 | 图纸/文档版本 |
| 通知 | 任务事件 | 状态变化触发通知 |
| 评论讨论 | 确认流 | 审批时留言 |
| 成本核算 | 任务属性 | 资源消耗记录 |

**原则**：分任务都是元能力的扩展，不是独立系统。

## 三、核心层设计（不变）

### 3.1 最小核心原则

```
┌─────────────────────────────────────────┐
│           Pclaw Minimal Core            │
├─────────────────────────────────────────┤
│                                         │
│   意图输入 → Agent 执行 → 结果确认     │
│       ↑              ↑              ↑   │
│       └──────────────┼──────────────┘   │
│                     │                   │
│               责任归属（唯一核心）       │
│                                         │
└─────────────────────────────────────────┘
```

**核心只有 3 件事**：
1. **意图引擎** - 理解人类目标
2. **Agent 编排** - 分解 + 调度执行
3. **确认流** - 人类审核节点 → 责任归属

### 3.2 核心接口定义

```typescript
// 意图输入
interface IntentInput {
  goal: string           // 人类目标
  constraints: string[]  // 约束条件
  deadline: Date        // 截止时间
  humanId: string        // 下令人
}

// Agent 执行
interface AgentExecution {
  taskId: string
  agentId: string
  plan: Task[]          // 分解的任务
  status: 'pending' | 'running' | 'completed' | 'failed'
  progress: number     // 0-100
  logs: ExecutionLog[]  // 执行日志
}

// 结果确认（责任归属关键）
interface ResultConfirmation {
  taskId: string
  agentId: string
  humanId: string       // 审核人
  result: any
  confirmStatus: 'pending' | 'approved' | 'rejected'
  confirmTime: Date
  liability: {
    human: string       // 最终责任人
    weight: number      // 责任比例 0-1
  }
}
```

## 七、责任与报酬体系

### 7.1 责任金字塔

```
┌─────────────────────────────────────────────────────┐
│                 责任金字塔                          │
│                                                     │
│                    法律责任                         │
│                   ┌─────────┐                       │
│                  /  人类    \   ← 最终责任主体       │
│                 /───────────\                       │
│                /   Agent     \  ← 执行责任          │
│               /────────────────\                    │
│              /    系统/设备     \  ← 环境责任       │
└─────────────────────────────────────────────────────┘
```

**关键原则**：
- Agent 犯错 = 人类审核失误
- 人类不确认 = 交付无效
- 重大决策必须人类在场

### 7.2 报酬体系

```
报酬 = Agent执行成本 + 人类审核价值 + 风险溢价

分配比例（示例）：
├── Agent 执行：60%
├── 人类审核：30%
└── 平台/组织：10%
```

## 十一、扩展性保障

| 原则 | 实现方式 |
|------|----------|
| **最小核心** | 核心层永不变，只有3个接口 |
| **模块化** | 插件系统，按需加载 |
| **配置化** | JSON/YAML 驱动 |
| **版本化** | API 版本管理 + 数据迁移 |
| **分布式** | P2P 连接 + 区块链（可选） |
| **可插拔** | 核心接口 + 实现分离 |

## 十二、关键哲学问题（持续思考）

1. Agent 能否"背锅"？ - 法律上不能，但系统内可追责
2. 人类审核的意义？ - 不仅是风控，更是责任归属
3. Agent 是否需要"休息"？ - 7×24 vs 维护窗口
4. Agent 之间是否平等？ - 专业分工 + 协作关系
5. 如何防止人类"甩锅"给 Agent？ - 强制确认流程
